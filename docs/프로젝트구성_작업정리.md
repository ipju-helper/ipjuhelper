## 1. 확정된 기술 스택

| 항목 | 선택 | 비고 |
|------|------|------|
| 프레임워크 | Next.js 14 (App Router) | 백엔드 역할 겸임 (API Routes) |

| 언어 | TypeScript | 전체 적용 |

## 2. 프로젝트 구조

### 모노레포 구성

```
IpJuHelper/
├── apps/
│   ├── web/                # 사용자 앱 (담당: 본인)
│   └── admin/              # 관리자 앱 (담당: 다른 개발자)
├── packages/
│   ├── shared/             # 공통 타입, 상수, 유틸 (담당: 본인)
│   └── supabase/           # DB 스키마, 마이그레이션 (담당: 본인)
├── turbo.json
├── pnpm-workspace.yaml
├── package.json
└── .gitignore
```

### 역할 분담

| 영역 | 담당 | 비고 |
|------|------|------|
| apps/web | 본인 | 사용자 앱 전체 |
| apps/admin | 다른 개발자 | 관리자 앱 전체 |
| packages/shared | 본인 | 공통 타입, 상수 — 본인만 수정 |
| packages/supabase | 본인 | DB 스키마 — 본인만 수정 |

### 앱 분리 이유

- 두 명이 각자 독립적으로 작업 → git 충돌 최소화
- 별도 배포 가능 (관리자 접근제어 분리)
- `turbo run dev --filter=web` / `--filter=admin`으로 본인 앱만 실행 가능

## 3. Git 전략

### 브랜치

```
main                           ← 단일 브랜치, 바로 머지
├── feat/web-auth              ← 기능 브랜치 (본인)
├── feat/web-inspection        ← 기능 브랜치 (본인)
├── feat/admin-dashboard       ← 기능 브랜치 (어드민 담당)
└── feat/admin-experts         ← 기능 브랜치 (어드민 담당)
```

- GitHub 레포 1개 (모노레포)
- `main` + 기능 브랜치, PR 없이 바로 머지
- develop 브랜치 없음
- `packages/`는 본인만 수정하므로 충돌 가능성 없음
- 브랜치 네이밍: `feat/web-*`, `feat/admin-*`, `feat/shared-*`

## 4. DB 스키마 설계 (CSV 기반)

### Enum 정의

```
case_type     : self, self_review, expert_visit
case_status   : draft, in_progress, submitted, completed
self_result   : normal, issue, not_checked
issue_level   : low, medium, high
shot_type     : overview, closeup, detail
expert_status : pending, approved, rejected
user_role     : user, expert, admin
category      : wall, floor, window, door, ceiling, kitchen, bathroom, balcony, electric, ventilation, common
```

### 테이블 목록

| 테이블 | 설명 | 기반 CSV |
|--------|------|----------|
| profiles | 사용자 프로필 (Supabase Auth 확장) | — |
| experts | 전문가 등록 정보 | — |
| cases | 점검 케이스 | cases_common.csv |
| checklist_master | 체크리스트 마스터 (52개 항목) | self_checklist_master.csv |
| self_inspection_results | 셀프 점검 결과 | self_inspection_results.csv |
| issue_photos | 하자 사진 | self_issue_photos.csv |
| issue_summary | 하자 요약 | self_issue_summary.csv |
| expert_schedules | 전문가 일정 | — |

### 주요 테이블 컬럼 (CSV 기반)

#### cases (cases_common.csv)

- case_id, case_type, user_id, expert_id
- customer_name, customer_phone, apartment_name, address
- building_number, unit_number, inspection_date, move_in_date
- status, memo, created_at, updated_at

#### checklist_master (self_checklist_master.csv — 52개 항목)

- item_key (PK), category, subcategory
- item_label, guide_text, help_text, photo_guide_text
- is_active, sort_order

#### self_inspection_results (self_inspection_results.csv)

- id, case_id (FK), category, subcategory
- item_key (FK → checklist_master), item_label
- result (normal/issue/not_checked), issue_level
- user_note, photo_required, is_flagged, sort_order

#### issue_photos (self_issue_photos.csv)

- file_id, case_id (FK), linked_item_result_id (FK)
- file_type, room_name, shot_type
- file_url, file_name, caption, uploaded_by_role, sort_order

#### issue_summary (self_issue_summary.csv)

- summary_issue_id, case_id (FK), linked_item_result_id (FK)
- room_name, category, issue_title, issue_description
- priority, photo_count

## 9. 참고: 주요 커맨드

```bash
# 전체 dev 서버
pnpm dev

# 특정 앱만 실행
pnpm dev --filter=web
pnpm dev --filter=admin

# 전체 빌드
pnpm build

# 특정 앱만 빌드
pnpm build --filter=web

# 타입 체크
pnpm type-check

# 의존성 설치 (루트에서)
pnpm install

# 특정 앱에 패키지 추가
cd apps/web && pnpm add 패키지명

# workspace 패키지 연결
cd apps/web && pnpm add @repo/shared --workspace
```
